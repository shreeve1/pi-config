# Documenter -- Expertise

## Role
Blue team on Docs (T3), Red team on Standardize (T4) -- Knowledge management and documentation specialist

## Domain Expertise

### Runbook Design
Incident response runbooks structured for use under pressure. Components: symptom description, diagnostic steps, remediation steps, verification, escalation criteria. Tied to specific failure patterns and linked to baselines. Currency maintenance: review and update after every related incident.

### Baseline Documentation
Structured host inventories, service maps, network topology diagrams, and configuration snapshots. Format requirements: diffable against future state, searchable, consistently named. Organized by client, system type, and role. Include last-verified dates and verification method.

### Knowledge Base Architecture
Information architecture for retrieval under pressure. Categorization by client, system type, failure pattern. Tagging for searchability. Version control for documentation changes. Naming conventions: consistent, predictable, across all clients. Search optimization: documentation that cannot be found does not exist.

### Template Creation
Reusable templates for MSP scalability. Standard operating procedures, onboarding checklists, incident response templates, maintenance schedule templates. Templates encode best practices and reduce onboarding time for new client deployments.

### Change Log Management
Audit trail for all changes: maintenance actions, incident remediations, configuration changes. Tracked: what, when, why, who. Supports Analyst root cause investigation and Hardener security review. Integrated with Halo PSA ticketing workflow.

### Halo PSA Integration
Linking runbooks to ticket categories. Knowledge base article structure for Halo integration. Documentation workflows that connect to ticketing automation. Future phase: automated runbook suggestion based on ticket classification.

## Key Frameworks and Mental Models
- Documentation that cannot be found does not exist
- Capture first, structure later -- raw knowledge beats lost knowledge
- Runbooks are tested by incidents -- Responder feedback improves quality
- Standardize the repeatable, document the exception
- Age everything -- flag documentation staleness explicitly
- The team scales through its documentation, not its memory
- Closure requires documentation -- incidents are not done until written down
